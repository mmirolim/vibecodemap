# Project DSL and renderer view model

The project DSL and generic view-model JSON are complementary. They solve
different problems and should not compete as two sources of truth.

## Project DSL: durable intent and evidence policy

The committed `*.project.vcm.yaml` is designed for humans and AI agents to read,
review, and edit. It records information that must survive regeneration:

- repository scope and analyzer policy;
- declared architecture and editable cluster membership;
- expectations, corrections, trust boundaries, and review candidates;
- metric meanings and renderer profiles;
- evidence and provenance.

It is semantic, not geometric. It should say that a district represents an
application service boundary, not that its plate starts at an exact WebGL
coordinate.

## View-model JSON: derived renderer input

The generic view model is a deterministic projection generated by the Go core
after it composes the project DSL with structural, quality, runtime, coverage,
and adapter evidence. It is optimized for rendering and interaction:

- the active decomposition and level-of-detail aggregates;
- normalized metric values plus unknown/freshness state;
- building forms, ordered bands, ports, security markers, and labels;
- aggregate district roads, typed lanes, assigned gates, and bounded feeders;
- source-navigation targets and inspectable evidence references;
- complete legends for every active visual channel.

It can be JSON embedded in a standalone HTML file. It is cacheable and safe to
regenerate. Hand-editing it is equivalent to editing a compiled artifact: the
next composition replaces it.

```text
repository + tool evidence + project DSL
                    │
                    ▼
              Go composition
                    │
                    ▼
       renderer-neutral view-model JSON
                    │
                    ▼
          Three.js browser presentation
```

The browser renderer does not infer architecture or silently recompute quality.
It lays out and presents already classified evidence. Exact coordinates may
change without changing the semantic model.

## Relation interpretation

A static import or type reference is topology: it can explain coupling,
clustering affinity, and why a utility is shared. It is not runtime
communication. The model therefore keeps relation family and execution state
separate:

- `import` + `static` is a source dependency;
- `call` + `sync` is blocking runtime behavior;
- `event` + `async` is asynchronous behavior;
- state/storage families identify persistence access.

The system-city prototype hides static imports in its default **Behavior** road
view and exposes them in an explicit **All + static dependencies** view. This
prevents an import such as “Flask API gateway uses shared model types” from
looking like observed network traffic while preserving its architectural value.

## Why not render the YAML directly?

Rendering directly from the project manifest would push composition,
cross-model joins, normalization, filtering, and aggregation into browser code.
That would duplicate Go validation rules and make visual output dependent on
hidden renderer inference. The derived view model keeps the DSL editable and
the renderer simple enough to test independently.
